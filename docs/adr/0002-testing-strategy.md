# ADR-0002: Testing strategy for a generated Upjet provider

**Theme:** 02 · Testing & quality · **Status:** Accepted · **Date:** 2026-07-15 · **Realises:** D-005

> **Signed off** by the operator on 2026-07-15 (approved as-is). Governs epic **E2**.

## Context

`provider-gridscale` ships **zero tests**. We need a strategy that raises confidence to a
professional OSS standard — but the obvious model (copy the sibling repos) does not transfer
cleanly, and the reason matters.

**The generation boundary changes what is worth testing.** kollect and mkurator are Kubebuilder
operators that **hand-write their reconcile logic**; their test suites are dominated by Ginkgo +
`controller-runtime` **envtest** reconcile tests, with **mockery** mocking the external system. That
tests *their own code*. In this provider the controllers (`internal/controller/**`), API types
(`apis/**/zz_*.go`) and CRDs (`package/crds/**`) are **generated by Upjet** from the terraform
provider. Unit/envtesting them would be testing Upjet, not us — high cost, ~zero signal, and it
breaks every regeneration.

**Two things investigated closely:**

1. **What actually ports from kollect/mkurator?** Only the tiers that don't depend on hand-written
   reconcilers:
   - **CRD schema golden-contract tests** — both repos extract each CRD's OpenAPI fragment and diff
     it against a checked-in golden (`test/schema/contract_test.go`, `UPDATE_GOLDEN=1` to refresh).
     No cluster, no envtest, milliseconds to run. For a *generated* provider this is the
     highest-ROI tier: it catches CRD-generation drift (a bad override, an Upjet bump changing
     validation) before it reaches users.
   - **Unit tests on hand-written support code** — kollect/mkurator unit-test `internal/mqadmin`,
     `internal/collect`, etc. Our equivalent is the small `config/` surface.
   - **Coverage + CI scaffolding** — `codecov.yml`, coverage floor scripts, task/CI wiring — portable
     as mechanics, but the *floor must be scoped to hand-written packages* (a repo-wide floor is
     meaningless when generated code is 95% of the tree).

2. **Chainsaw vs uptest?** This is largely a false choice. **uptest already *is* Chainsaw**: it is a
   manifest-to-Chainsaw compiler that renders each example CRD into a Chainsaw `Test`
   (apply → import → update → delete, with JMESPath assertions) and executes it. This repo already
   has the machinery wired — `make uptest`/`make e2e`, `CHAINSAW 0.2.13` installed and passed to
   uptest, `cluster/test/setup.sh` creating the credential secret and both `ProviderConfig` variants,
   and `.github/workflows/e2e.yaml`'s `/test-examples` trigger (secret `UPTEST_CLOUD_CREDENTIALS` =
   gridscale token). Standalone Chainsaw suites (as in kaddy) prove behaviour *above* the controller
   — Compositions, Kyverno policy, observability — which are **platform** concerns, not this
   provider's. Writing hand-rolled Chainsaw CRUD tests would duplicate what uptest generates.

## Decision

Adopt a **four-tier, provider-shaped** taxonomy (recorded in `openspec/config.yaml`). Every REQ
carries exactly one `**Test:**` (artifact) + one `**Verify:**` (command) + a `**Level:**`.

| Level | Scope | Tooling | Runs | Cluster/creds |
| --- | --- | --- | --- | --- |
| **U** unit | Hand-written `config/` + `hack/` helpers | `go test` (stdlib + testify) | every push (`make test`) | none |
| **M** meta/contract | **CRD golden-contract** (flagship) + structural invariants (`make check-diff`, `crddiff`, CI-job-present, lint) | `go test ./test/schema/…` + shell asserts | every push | none |
| **E** e2e | Example-as-test against the live gridscale API | **uptest** (renders Chainsaw) | `/test-examples`, nightly | kind + gridscale token |
| **D** doc-sync | Generated docs match source (E4) | `crd-ref-docs`/examples diff | every push | none |

**Concretely for E2:**

- **U** — unit-test the hand-authored surface, TDD-first. The genuine target is the **custom
  external-name logic**: `config/external_name.go` defines `idWithStub()`, which overrides
  `GetExternalNameFn` to map tfstate → provider UUID (`config.IDAsExternalName`). That is real,
  isolatable logic — assert `{"id":"<uuid>"} → "<uuid>"` and the include-list (`.+`) shape.
  - The two `config/gridscale.go` Kind overrides (mysql8 → `MySQL8`, storage_import → `StorageImport`)
    are **not** given their own U-story: they're already guarded by compilation itself (a broken
    override yields Go kind `"0"` / reserved word `import` and `make build`/`make generate` fails) and
    by the CRD-contract tier. An *optional* cheap assertion that the built provider config carries
    those Kinds can ride along, but it has low marginal value over build + M.
- **M** — adopt the kollect/mkurator **CRD golden-contract** test, **re-pointed at this repo's layout**:
  extract the OpenAPI schema from `package/crds/**` (Upjet output; the siblings read Kubebuilder's
  `config/crd/bases/`) and diff against checked-in goldens; `UPDATE_GOLDEN=1` to refresh. **Scope the
  golden set** to a representative + override-affected subset (e.g. MySQL8, StorageImport, Server,
  Network) — golden-ing all **69** CRDs (32 resources × 2 scopes + config) would be enormous and noisy
  on every regen. Keep the existing structural guards (`make reviewable`, `make check-diff`) as M REQs.
- **E** — this is the **highest-effort tier, not a freebie.** The uptest *machinery* exists, but
  `examples-generated/` carry Terraform-style refs (`${gridscale_ipv4.terra-ipv4-test.id}`) and
  placeholder UUIDs (isoimage, etc.). Turning them into runnable tests means curating a **smoke
  subset** (server, network, storage, ipv4, sshkey), wiring refs via the uptest datasource, and
  running against a **live gridscale lab project** — so E is gated on lab access. Keep behind
  `/test-examples` + nightly (real API cost + rate limits); document the creds contract.
- **Coverage** — codecov upload for visibility; a **floor scoped to the hand-authored packages
  `./config/...` and `./internal/clients/...`** (start ~70%, ratchet up), **not** repo-wide. Generated
  packages (`internal/controller/**`, `apis/**/zz_*.go`, `package/crds/**`) are excluded from the floor.
  > **Updated by [D-007](../../agent-context/decisions.md) (2026-07-16):** the floor originally covered
  > only `./config/...`. It now also includes `./internal/clients/...` — the audit-critical credential
  > surface (`TerraformSetupBuilder`/`resolveModern`/`resolveLegacy`/`buildProviderConfiguration`, which
  > map a `ProviderConfig` + `Secret` into the Terraform provider setup). This is a hand-authored package,
  > so the generation-boundary exclusion does not apply to it; the floor still stops at the generated tree.

**Explicitly out of scope (and why):**

- ❌ **envtest / Ginkgo reconcile tests** — controllers are generated; nothing of ours to exercise.
- ❌ **mockery / interface mocks** — no hand-written business interfaces to mock.
- ❌ **Repo-wide coverage floor** — generated code would dominate and make the number a lie. (The
  floor stays scoped to the hand-authored packages `./config/...` + `./internal/clients/...`; see D-007.)
- ❌ **Standalone Chainsaw suites** — uptest already renders Chainsaw for CRUD; hand-rolled Chainsaw
  only pays off at the Composition/policy layer, which is kaddy's E6g-S03, not this provider.

## Consequences

- The test surface stays honest: we test *what we wrote* (config overrides, examples) and *what we
  ship* (CRD schemas), not machine-generated glue.
- **M-level CRD golden tests** give fast, cluster-free regression protection against Upjet bumps and
  bad overrides — the single best confidence-per-second tier for a generated provider.
- **E** reuses infrastructure that already exists in the repo; the work is curating examples and
  wiring the datasource/secret, not building a harness.
- Coverage numbers are meaningful because they're scoped to the hand-written surface.
- We diverge from the sibling repos' Ginkgo/envtest house style — accepted, because their style is a
  consequence of hand-written controllers we don't have.

## Counterpoints considered

- *"Match kollect/mkurator: Ginkgo + envtest for consistency."* Rejected: it would test generated
  controllers (i.e. Upjet), cost a lot, and churn on every regen. Consistency of *value* (golden
  contracts, coverage discipline, TDD on hand-written code) matters more than consistency of *tool*.
- *"Add standalone Chainsaw suites like kaddy for richer e2e."* Rejected for CRUD (uptest already
  emits Chainsaw); revisit only if this repo ever grows Compositions/XRDs or admission policy.
- *"Set an ambitious repo-wide coverage floor (80–90% like the siblings)."* Rejected: meaningless
  over a generated tree. A scoped `config/` floor is the honest analogue.
- *"Skip uptest for v1 — it needs real creds and a lab project."* Rejected: without at least a smoke
  E-tier we can't claim the provider actually provisions anything; keep it gated but present.
